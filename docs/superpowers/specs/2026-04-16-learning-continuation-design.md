# 学习专题续写课时设计

**日期：** 2026-04-16
**状态：** 已按当前对话确认，等待你审阅后进入实现计划

## 背景

当前学习 CMS 已经支持：

- 在后台创建和编辑学习专题
- 使用 AI quickstart 一次性生成一个新专题及其首批课时草稿
- 在公开学习中心读取数据库专题，并兼容静态专题兜底

但它还不支持一个更符合长期运营的工作流：当某个专题已经存在若干课时后，管理员希望继续沿着这条学习线追加后续课时，而不是每次重新新建专题，或手动填写完整 JSON 表单。

用户已经明确希望：

1. 这次只做学习中心，不涉及小说。
2. 总学习轴保持固定，不让 AI 改顶层结构。
3. 本期只给现有专题追加新课时，不新建专题。
4. 生成流程采用双阶段：
   - 先生成“续写方案 / 课时大纲”
   - 再在管理员确认后生成正式课时草稿

## 目标

在现有学习 CMS 基础上，新增一套“专题内 AI 续写课时”能力，满足以下目标：

1. 管理员可以在现有专题详情页内，直接为该专题补充后续课时。
2. AI 不直接静默落库，而是先给出 1 到 3 节候选续写方案。
3. 管理员确认方案后，再生成完整课时草稿并写入数据库。
4. 所有新课时默认保存为 `draft`，不自动发布。
5. 过程要有明显的阶段反馈、日志和结果摘要，避免“看不出是否开始 / 卡住”的体验。
6. 不破坏现有手动创建课时表单、前台学习页、搜索、聊天和 quickstart 能力。

## 非目标

本期明确不做：

1. 不支持为静态专题直接续写；只支持数据库中的 managed track。
2. 不支持新建专题；只支持给现有专题追加课时。
3. 不支持改写、覆盖、批量重排已有课时。
4. 不支持把“续写方案”持久化为数据库历史记录。
5. 不做后台任务队列、断点恢复或跨刷新恢复。
6. 不引入新的 AI 配置体系；继续复用后台现有的 `baseUrl / apiKey / model` 本地保存方式。

## 已确认的产品决策

### 一、入口位置

入口放在现有专题详情页：

- `src/pages/admin/learning/[id].astro`

在当前“添加新课时”区域附近升级为一个更完整的 `AI 续写课时` 工作区。现有手动建课时表单保留，但降级为折叠或次要入口。

### 二、交互模式

采用双阶段工作流：

1. **生成续写方案**
   - 读取当前专题、已有课时和可选补充方向
   - 先生成 1 到 3 条下一批课时方案卡片
2. **生成课时草稿**
   - 管理员确认或轻微修改方案后
   - 再逐节生成完整课时内容并写入数据库

### 三、输入最小化

管理员在一期只需要提供最少输入：

1. 生成几节：默认 `2`，可选 `1` 到 `3`
2. 可选补充方向：一句话，例如“补缓存实战”“补更通俗的 TCP 进阶内容”

不要求管理员手写大纲、顺序或完整课时字段。

### 四、输出默认策略

1. 新课时总是追加到当前专题最后面，`order` 从现有最大值后继续递增。
2. 每节新课时一律先保存为 `draft`。
3. 生成失败时，不回滚已经成功写入的草稿，而是返回部分成功结果。

## 方案比较

### 方案 A：在专题详情页内嵌续写面板

优点：

- 路径最短，管理员本来就在当前专题页管理课时
- 能直接复用专题上下文和现有课时列表
- 开发成本最低，符合本期只做“续写”而非“新建专题”的范围

缺点：

- 专题详情页会变得更重
- 页面内状态管理会比纯表单复杂

### 方案 B：独立续写工作台页

优点：

- 可以做更完整的日志、阶段流和预览
- 后续更容易扩展历史记录与批量生成

缺点：

- 增加新的路由和跳转
- 对当前一期范围而言偏重

### 方案 C：全站通用生成中心

优点：

- 未来学习、小说等都可复用

缺点：

- 明显超出本期需求
- 会把本来简单的“给专题补课时”动作做复杂

### 结论

本期采用 **方案 A：在专题详情页内嵌续写面板**。

## 页面设计

## 一、页面结构

专题详情页保留三块核心信息：

1. 专题元信息编辑区
2. `AI 续写课时` 工作区
3. 现有课时列表

原有“添加新课时”表单保留，但移到 `手动创建课时` 折叠区，作为 AI 续写之外的备用入口。

## 二、AI 续写课时工作区

工作区分为四个阶段：

### 1. 选择范围

展示：

- 当前专题标题
- 已有课时数
- 下一课序号

输入：

- 课时数量选择器：`1-3`，默认 `2`
- 补充方向输入框：可为空

动作：

- 主按钮：`生成续写方案`

### 2. 方案预览

服务端返回 1 到 3 张方案卡片，每张卡展示：

- 预计序号，例如 `第 6 课`
- 课时标题
- 一句话说明为什么现在补这节
- 预计时长
- 目标摘要
- 关键词摘要

管理员在方案阶段可进行轻量调整：

- 删除单张方案卡
- 修改标题
- 修改简短描述

方案阶段动作：

- `重新生成方案`
- `生成课时草稿`

### 3. 生成进行中

在生成草稿时展示明显的阶段条和日志，而不是只显示一个转圈：

- 读取专题上下文
- 生成续写方案
- 生成第 1 节草稿
- 生成第 2 节草稿
- 生成第 3 节草稿

日志示例：

- 正在分析最近 3 节课的衔接关系
- 已完成第 1 节草稿校验，正在写入数据库
- 第 2 节草稿已保存，正在生成第 3 节

忙碌态要求：

- 正在请求期间禁用重复提交
- 主按钮文案变为对应进行中状态

### 4. 结果摘要

成功时展示：

- 成功生成了几节草稿
- 每节课的标题
- 每节课对应的后台编辑链接
- 一个 `继续追加下一批` 按钮

部分失败时展示：

- 哪几节成功
- 哪几节失败
- 成功项保留编辑入口
- 失败项给出 `重新生成失败项` 或 `返回方案阶段` 的入口

## 接口设计

新增两个后台 API，均挂在专题下：

### 1. `POST /api/admin/learning/:id/continuation-plan`

作用：

- 读取当前专题及最近若干课时
- 拼装续写上下文
- 调用 AI 生成下一批课时方案
- 返回轻量方案卡片，不落库

请求体建议：

```json
{
  "count": 2,
  "direction": "补缓存与性能优化内容"
}
```

返回体建议：

```json
{
  "track": {
    "id": "track_123",
    "title": "计算机网络"
  },
  "startingOrder": 6,
  "plans": [
    {
      "order": 6,
      "title": "HTTP 缓存到底在缓存什么",
      "description": "衔接请求生命周期之后，继续补齐浏览器与服务端缓存协作。",
      "estimatedMinutes": 14,
      "objectives": ["理解强缓存与协商缓存", "知道常见缓存头的作用"],
      "keywords": ["http", "cache", "etag"],
      "animation": "request-path",
      "heroSummary": "从浏览器再次访问同一资源的真实路径切入。"
    }
  ]
}
```

### 2. `POST /api/admin/learning/:id/continuation-drafts`

作用：

- 接收管理员确认后的方案卡
- 逐节生成完整课时草稿
- 调用现有校验与建课时逻辑写入数据库

请求体建议：

```json
{
  "direction": "补缓存与性能优化内容",
  "plans": [
    {
      "order": 6,
      "title": "HTTP 缓存到底在缓存什么",
      "description": "衔接请求生命周期之后，继续补齐浏览器与服务端缓存协作。",
      "estimatedMinutes": 14,
      "objectives": ["理解强缓存与协商缓存", "知道常见缓存头的作用"],
      "keywords": ["http", "cache", "etag"],
      "animation": "request-path",
      "heroSummary": "从浏览器再次访问同一资源的真实路径切入。"
    }
  ]
}
```

返回体建议：

```json
{
  "message": "Generated 2 draft lessons, 1 failed.",
  "created": [
    {
      "id": "lesson_1",
      "title": "HTTP 缓存到底在缓存什么",
      "adminUrl": "/admin/learning/track_123/lessons/lesson_1"
    }
  ],
  "failed": [
    {
      "order": 7,
      "title": "缓存命中与性能观测",
      "reason": "Lesson validation failed."
    }
  ]
}
```

## 数据与持久化设计

## 一、本期不新增数据库表

本期不为“续写方案”新增持久化表。

原因：

1. 方案只是草稿前的临时中间态
2. 当前最重要的是把双阶段流程跑通
3. 不落库可以显著降低复杂度和回归风险

因此：

- `continuation-plan` 返回的数据只在当前后台页面前端状态中保存
- 只有 `continuation-drafts` 成功后，才真正写入 `learning_lessons`

## 二、继续复用现有领域模型

直接复用：

- `src/lib/learning/repository.ts`
- `src/lib/learning/validation.ts`
- `src/lib/learning/types.ts`

重点复用：

- `createLearningLesson`
- `validateLearningLessonInput`
- 现有 `ManagedLearningSubject` / `ManagedLearningLesson` 数据结构

本期只需要在应用层增加一个“续写方案”的临时类型，例如：

```ts
type LearningContinuationPlanItem = {
  order: number;
  title: string;
  description: string;
  estimatedMinutes: number;
  objectives: string[];
  keywords: string[];
  animation: LearningAnimationKind;
  heroSummary: string;
};
```

这个类型只在 API 和前端页面状态里流转，不入库。

## AI 生成链路设计

## 一、上下文拼装

服务端自动拼装续写上下文，不要求管理员手工准备大纲。

专题级上下文：

- 标题
- 描述
- 阶段
- learningPath
- conceptOverview

课时级上下文：

- 当前专题已有课时总数
- 最近 2 到 4 节课的标题
- description
- objectives
- keywords
- heroSummary
- chatContextSummary

结构约束：

- 下一课序号从 `max(order) + 1` 开始
- 一次最多生成 3 节
- 方向补充词作为可选附加约束

## 二、AI 函数拆分

在现有 `src/lib/ai/learning.ts` 上新增两个能力：

### 1. `generateLearningContinuationPlan(...)`

负责：

- 根据现有专题与最近课时
- 产出下一批课时的轻量方案卡片

它不生成完整正文、误区、小测和 sections。

### 2. 继续复用 `generateLearningLessonDraft(...)`

在管理员确认方案后，继续使用现有课时草稿生成器：

- 输入从 quickstart 的 outline 改为 continuation plan item
- 输出仍是完整 lesson draft

这样可以最大化复用当前 AI 学习写作链路，而不是再造一套草稿生成器。

## 三、slug 与顺序策略

1. `order` 不由 AI 自由决定，服务端按当前最大序号顺延。
2. `lessonSlug` 服务端根据标题自动生成。
3. 如果 slug 冲突，则自动追加后缀。
4. AI 不能插入或覆盖已有序号。

## 页面状态流

前端状态建议拆成几个明确阶段：

- `idle`
- `planning`
- `plan-ready`
- `drafting`
- `completed`
- `partial-failed`
- `failed`

并维护以下关键状态：

- `count`
- `direction`
- `planItems`
- `currentStep`
- `logs`
- `createdDrafts`
- `failedDrafts`

状态迁移建议：

1. `idle -> planning`
2. `planning -> plan-ready`
3. `plan-ready -> drafting`
4. `drafting -> completed`
5. `drafting -> partial-failed`
6. `planning/drafting -> failed`

## 边界与保护

1. 仅允许已存在于数据库的专题使用续写功能。
2. 若专题当前 `0` 节课，则禁用 AI 续写，并提示先建立第一节基础课时。
3. 一次最多生成 `3` 节，默认 `2` 节。
4. 所有新课时一律先保存为 `draft`。
5. 不覆盖已有课时，不修改已有课时正文。
6. 不允许跨专题复用这次生成的方案。
7. 页面刷新后，未确认的方案丢失；已成功入库的草稿保留。

## 错误处理

### 一、方案阶段失败

当 `continuation-plan` 失败时：

- 保留 `count` 和 `direction`
- 显示“方案生成失败”的明确提示
- 不清空用户已填内容

### 二、草稿阶段失败

当 `continuation-drafts` 部分失败时：

- 已成功创建的课时保留
- 失败项单独列出
- 前端显示“部分成功”的结果摘要
- 允许用户返回方案阶段重新处理失败项

### 三、脏数据保护

在两阶段分别做校验：

1. 方案阶段做轻校验：
   - 标题
   - 描述
   - 时长
   - objectives
   - keywords
   - animation
   - heroSummary
2. 草稿阶段做正式 `validateLearningLessonInput`

### 四、慢请求与卡住问题

前端必须提供：

- 明确的阶段条
- 实时日志
- 按钮忙碌态
- 错误提示

避免只显示静态按钮和无反馈加载状态。

- `npm test`

## 实现范围建议

为了控制本期复杂度，建议实现顺序如下：

1. 先做 continuation plan 的服务端能力
2. 再做 continuation drafts 的服务端能力
3. 然后接专题详情页前端状态流
4. 最后整理文案、忙碌态和结果摘要

这样可以先确保领域与接口成立，再补齐交互体验。

## 总结

本期最终设计结论是：

1. 在现有专题详情页内增加 `AI 续写课时` 工作区
2. 只支持给现有 managed track 追加后续课时
3. 采用“先生成续写方案，再确认生成草稿”的双阶段流程
4. 不新增数据库表，方案只保存在前端临时状态中
5. 继续复用现有学习 CMS 的 repository、validation 与 lesson draft 生成能力
6. 把明显的阶段反馈、日志、忙碌态和结果摘要作为一期必需体验，而不是附加优化
