## 为什么静态博客也要做性能检查

即使站点本身已经足够轻量，如果首页信息过多、资源组织混乱，依然会影响首屏体验。性能检查不是为了追求极限分数，而是为了避免明显问题。

### 先看页面结构

我通常会先确认首页区块是否过多、标题层级是否合理、文章列表是否有重复信息。这些问题往往比图片压缩更早影响体验。

### 再看资源输出

构建后观察输出结果，确认没有异常膨胀的脚本或样式体积。对于个人博客来说，保持简单往往就是最有效的优化。

## 一份够用的上线前清单

1. 首页是否仍然聚焦最重要内容
2. 文章页目录是否清晰可读
3. 搜索与归档页是否能快速定位内容
4. RSS、sitemap、canonical 是否输出正确
5. 移动端下导航和卡片是否正常换行

### 检查构建结果

```bash
npm run build
```

如果构建日志里所有页面都正常生成，接着再用预览或线上环境抽查核心页面。

## 不要为了优化而复杂化

很多个人博客的性能问题，不是因为技术选型不对，而是因为后续迭代引入了太多并不必要的资源。保持组件简单、页面克制，往往能自然得到不错的结果。

## 结语

性能不是最后才补的一层包装，而是每次上线前都可以快速过一遍的基础检查项。
